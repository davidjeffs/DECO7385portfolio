Success Criteria

As discussed in the reflective report, the user-centric design was switched out for one that is activity-centred. The evaluation which follows is an attempt to reconcile the data gathered on this success criteria with that gathered at the exhibition.

General Success Criteria

Users complete goal 80% of time

FAIL - Whilst there was no quantifiable data gathered, users were observed to leave The Social Dial after a short time.

PASS - users were observed to complete tasks on the Twitter Tapper.

Value on communication technologies

FAIL - Users comments regarding The Social Dial were largely regarding the interaction rather than a reflection. Some comments were made about users preferring phones.

PASS - users made passing comments regarding the difficulty of posting a message

Users use positive terms

PASS - Positive terms were used for both The Social Dial and the Twitter Tapper however it is thought that this feedback needs to be taken critically as people in exhibitions cannot be assumed to be typical user testers.

The Social Dial Success Criteria

No more than 3 misdials

PASS - There were only a few instances in which the user was observed resetting the machine, or using number 9 to listen to the options again, or 0 to go back.

Messages translated with 80% accuracy

PASS - The messages posted were quite short, however from the messages posted on the Facebook wall, and messaged to others, we can see that they make sense.

Voice prompts listened to again

PASS - Users were rarely observed to relisten.

Twitter Tapper Success Criteria

Navigation with no more than 20% mistaps

FAIL - Examining the data received on Twitter we can see that the majority of people were unable to type a morse code message.

60% completion of messages

PASS - Based on the timings of the Twitter posts, we can see that most users were able to complete their message and have it posted as one rather than a split post.

Discussion

Whilst the success criteria presented were not satisfactory to evaluate the success of the project, we can examine the feedback from an activity-centred point of view. The reason for this can be seen in the usability centred,and user-centric approach from the success criteria. A more encompassing approach is required, which is why the reflective report argues that an activity-centric approach allows for speculative designs to be analysed in this way.

If we understand that there are contradictions between the various nodes of the activity centred approach we can examine whether these contradictions have been achieved in the concepts produced here.

Based on the observations provided at the exhibition there was some contradictions occuring between the traditional activity model, and the futuristic one proposed by The Social Dial. This contradiction is largely based on the different rules effecting the interaction of users with the concept. However it may not be possible to state that these contradictions specifically caused a reflection on the idea that communication is easier with smart phones.

Along the same lines of the contradiction we can certainly see that there was mismatch between the object and subject for younger generations who were unaware of the exact interaction with the rotary phone. Whilst these users did not usually use the phone for long, this may be a result of the frustration caused. A deep follow up interview would be required for this social grouping before any full assertions can be made.

Following Up

Whilst some contradictions may have occured, and they may have caused reflection there are several areas of the design process and the concept that need to be addressed by future research.

As discussed in the reflective report, the process undertaken was user centric which from the start effected the design direction. By taking an activity centred approach a more thorough speculative design can be achieved. However that does not mean users are not required. Users are one main node on the activity centred model.

There is also a need to run the agile development process with a tighter consideration of deadlines. There were aspects of the design process which were held up by difficulties in the development process. Whilst other design activities were undertaken to fill the void, these activities were not as information producing as ones that were aimed for with a development process on time.

However that is also a double edged sword. The beginning of the process was held up by limitations put on the design process.

Future Directions

Whilst the concept produced was able to satisfactory explore the research space, there are still largely areas to explore. These include a deeper exploration of the context being designed for, which is the home environment.

There is also scope to further explore detailed social media interactions rather than the main generalised ones used on the The Social Dial and Twitter Tapper.

Further research can also be undertaken into graphically based social media networks.

Finally, whilst this project was speculative in nature, there is room for practical application. Two main areas are providing accessibility to modern communication methods to older adults suffering from cognitive decline who would benefit from the familiarity provided by a rotary phone, but also the social engagement. Second there is possibility to assist the vision impaired in accessing graphic oriented social media networks through the use of image recognition algorithms.

PHYSICAL CONSTRUCTION

The Social Provocateurs attempted to use existing form factors as part of their design to maintain the user experience. Whilst team member William Hayward was able to maintain an understanding of the telephone, David Jeffs undertook the design and construction of the morse code machine.

Morse Code construction

The morse code machine was constructed using dark wood and brass as a method of maintaining traditional construction techniques. This was to skew the user experience towards a morse code context.

Construction Activities

Physical

  1. Duplication of measurements from plastic morse code machine
  2. Constructing back-board for morse alphabet (includes hanger for headset)
  3. Sand and polishing of base board.
  4. Addition of brass corners and brackets to morse code machine
  5. Lacquer of morse base.

Ardunio

  1. Soldering of headphones and button into arduino, and testing.
  2. Programming of arduino (by Will)
  3. Connection to Twitter (by Will)

Reasons behind some of the physical

Aesthetics

  1. The aesthetics were improved such that the situated cognition would kick in and draw user into the context of morse code
  2. This was linked to the use of brass and wood which are traditional construction materials of morse code machines.
  3. Julien added a Twitter symbol to the morse code button such that there was visibility of Twitter on the machine

Structure

  1. The pin board was placed directly in front of the user to minimise head movement.
  2. The headphones were added to be the least intrusive to the morse code (coming out the back), yet be usable in order to focus user attention to the task at hand.

Construction

The construction took place over several weeks with various stages.

Programming

The Social Provocateurs concepts invovled various bits of programming. Whilst Will Hayward produced much of it, David Jeffs produced the python framework for which The Social Dial was based. This was a match the UML diagram produced in earlier activities.

Python framework

The Python framework allows the user to walk through The Social Dial and access all of the functions. Several technical decisions were made as part of this.

Some technical decisions

The key decisions

  1. For the text-to-speech initially the popular Pyttsx for python was used. THe framework was developed around this, however with the introduction of threading it was discovered that threading and pyttsx are incompatible on Mac. Thus a switch was made to Google text-to-speech which was slower.
  2. Threading was programmed in to allow for multiple functions to run simultaneously. These functions include dialing, playing speech, and access the Facebook API developed by Will.
  3. Python was used simply due to its simplistic nature and ease of connection to the variety of frameworks that we were accessing.
  4. To allow for users to spell their name a change was made to the speech address values. Alphabet letters were added to prevent users from spelling out words.
  5. To allow users to state the word 'space' for the gap between their name, .replace() was used
